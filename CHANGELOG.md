## Unreleased

## 2.0.2 — 2026-06-23

### Added
- **`--ssh` rebuilt into a threat-hunt / DFIR analyzer with evidence-backed, ATT&CK-mapped detections.** It was largely an inventory/stats dump whose only behavioral checks needed decrypted traffic, so a textbook SSH brute force (11 back-to-back guessing sessions) produced *no* detection at all. SSH user-auth is encrypted, so the analyzer now **infers authentication outcomes from post-handshake traffic volume** (Zeek-style): each session is classified `established` / `failed-or-short` / `no-kex` from the bytes/packets exchanged after NEWKEYS, and a new **Auth Activity (Inferred)** table plus per-target detections surface brute force (`T1110`) and likely credential compromise (`HIGH` when failed-looking sessions are followed by an established one). New packet-observable detections, each carrying **severity, confidence, ATT&CK technique(s), and an Evidence list**:
  - **Terrapin (CVE-2023-48795)** — per-handshake assessment: a vulnerable cipher mode (`chacha20-poly1305@openssh.com`, or a CBC cipher with an `*-etm@openssh.com` MAC) negotiated **without** strict-KEX (`kex-strict-*-v00@openssh.com` absent from both KEXINITs) → `T1557`.
  - **SSH host key changed** for one service `(ip, port)` — possible MITM / key rotation (`T1557`).
  - **Egress / internal→external SSH** — tunneling / exfil / unsanctioned remote access (`T1048`/`T1572`/`T1021.004`).
  - **SSH on non-standard port** (`T1571`), **legacy SSH-1.x/1.99** offered (`T1040`/`T1557`), **SSH beaconing** (external = `warning`/C2, internal = `info`), **horizontal auth sweep** across many hosts (`T1110`/`T1046`), large-outbound **exfil** (`T1048`), and **scripted/library client & server** banners (reverse-shell/implant signal, `T1021.004`/`T1572`).
  - Detections and anomalies now render their ATT&CK IDs and evidence bullets; the consolidated `--threats` view inherits all of it.

### Fixed
- **`--http` "Top Clients" stamped every client with the *server's* hostname.** The `Host:` request header names the server (the request target), but it was being added to `client_host_counts[src_ip]` as well as `server_host_counts[dst_ip]` — so in a capture where many workstations hit one web/SQL host, every client row showed that server's name (e.g. `waraypvpsql10`) as if it were the client's own hostname. The Host header now maps to the destination only. A client IP shows a hostname only when that same IP is independently observed acting as a named HTTP server (proxy / dual-role / loopback); pure clients correctly show `-`. (`--tls` was reviewed and is correct — SNI is shown as "Top SNI targets" and as a per-conversation column tied to the server; `--quic` has no hostname table; `--dhcp` maps option-12 hostnames to the announcing client, which is correct; the `hostname.py` aggregator is already direction-aware.)
- **`--http` crashed on malformed request/referrer URLs.** A single request whose URL had an unbalanced IPv6 bracket made Python's `urlparse` raise `ValueError("Invalid IPv6 URL")`, aborting the entire HTTP analysis (reproduced on `Malcolm-PCAP/.../HTTP_1.pcap`). All `urlparse` calls now go through a non-raising `_safe_urlparse` wrapper.
- **`--ssh` host-key fingerprints were wrong for Diffie-Hellman group-exchange handshakes.** SSH message number 31 is overloaded — it is `KEX_DH_GEX_GROUP` (carrying the per-connection prime `p`, which sshd randomizes) under group exchange, but the parser always treated 31 as a host-key reply, so it hashed the prime and reported a *different* "host key fingerprint" on every connection to the same server. The parser now reads the host key from the correct messages (`31` for plain-DH/ECDH, `33` for GEX) and **validates that the blob begins with a real host-key algorithm name**, so group-exchange servers now yield one stable fingerprint (and the new host-key-change detector doesn't false-positive).
- **`--ssh` plaintext sections were noise.** SSHv2 carries no application data before encryption, yet the analyzer string-scanned the binary KEXINIT, dumping cipher/MAC/KEX **algorithm name-lists** (and their length-prefix bytes) into "Observed Plaintext" and matching the `ssh-rsa`/`ssh-ed25519` *algorithm names* as fake "SSH key material". Pre-encryption scanning was removed; plaintext/credential/key scanning now runs only on genuinely decrypted channel data, and the key-material pattern matches actual base64 key blobs. The blanket "Weak SSH algorithms" finding (offered ≠ negotiated, fired on nearly every OpenSSH handshake) was down-ranked to informational.

- **`--streams` no longer mislabels unestablished TCP flows as completed connections.** A "stream" in pcapper is any group of packets sharing a canonical 4-tuple, and (without `-established`) a record is emitted for every flow — including SYN-only attempts that never connected. The summary, however, reported `Complete Streams` / `Incomplete Streams` derived solely from TCP *reassembly* gaps; a SYN-only flow carries no payload, has no gaps, and was therefore counted as "Complete." On an egress-firewalled capture of nothing but blocked outbound SYNs (35k SYN/SYN-ECN packets, zero SYN-ACKs), this read as "11198 Complete Streams" — directly contradicting `--threats`, which correctly called the same flows "unanswered SYNs (SYN-ACK=0%)." `analyze_streams` already tracked handshake state (`established`/`synack_seen`/`syn_dir`) but discarded it. Now:
  - **Each `StreamRecord` carries a `conn_state`** derived from the handshake: `established` (full 3-way), `active` (bidirectional data but SYN not captured — capture started mid-session), `half-open` (SYN-ACK, no final ACK), `refused` (SYN→RST), `syn-only` (unanswered SYN — blocked egress/scans), `reset`, or `no-handshake`.
  - **The summary now leads with connection state** — `Established (3-way handshake)`, `Active (mid-capture, no SYN)`, and `Not Established` with a per-state breakdown — and prints an explicit note when nothing completed a handshake. Reassembly completeness is kept as a clearly separate `Reassembled Cleanly` / `Reassembly Gaps` metric.
  - **The stream table gained a `State` column**, the per-row reassembly cell shows `no-data` for payload-less flows instead of a vacuous `complete`, and the `-id` Selected Stream Metadata shows `Connection State`.

## 2.0.1 — 2026-06-16

### Changed
- **`--files` File Hashes section is now opt-in via `-hash`.** The MD5/SHA-256 "File Hashes (VirusTotal / threat-intel lookup)" table no longer prints by default; pass `-hash` to show it. `-hash` takes an optional filename: bare `-hash` lists hashes for all discovered files, while `-hash FILENAME` shows only the matching file(s) (the per-match SHA256/MD5/IMPHASH "Requested File Hashes" table) and suppresses the all-files table. `-hash FILENAME` also filters the **Discovered Files** table to the matching file(s), so the whole files view is scoped to that file. A capture path given after a bare `-hash` (e.g. `--files -hash capture.pcap`) is correctly treated as the target rather than being swallowed as the filename filter.
- **Removed every "… Risk Matrix" section across all analyzers.** The Risk Matrix tables (Domain, DNS, HTTP, TLS, FTP, ICMP, IPS, TCP, UDP, Beacon, MITRE, Files, Protocol, Service, Certificate, Health, Compromise, Host, Hostname, Timeline, ARP, DHCP, Threats, Exfil) and all of their backing logic were deleted — both the rendered output and the `risk_matrix` summary fields plus the enrichment/aggregation code that computed them. Analyst Verdicts, the "Deterministic … Security Checks" sections, and all other output are unchanged. (The unrelated "CTF Confidence Matrix" is retained.)
- **`--hostnames` drops the "Deterministic Hostname Security Checks" and "Hostname Risk Matrix" sections.** Both the output and the underlying logic (the `deterministic_checks`/`risk_matrix` summary fields and their enrichment computation) were removed. The Analyst Verdict, Protocol Evidence, Discovery Methods, and the hostname tables are unchanged.
- **`--hostnames` now honors the optional `-ip` filter** (and `-name`/`-port`/`-search`). With `-ip <addr>` the output is scoped to hostnames discovered for that IP instead of every host in the capture; with no filter it lists all discovered hostnames as before.

### Changed
- **`--ips` is now an IP-triage/visualization tool with geolocation enrichment.** The IP Host Details table gained a **Class / Role** column (internal/external × client/server/scanner) and a **Geo / ASN / Org** column. A new opt-in **`--ip-geo`** flag enriches public IPs via the free ip-api.com online lookup (country, ASN, org, and hosting/proxy/mobile flags) — e.g. an EK gate resolves to "Germany · AS24940 Hetzner [hosting]" and a C2 high-port host to "Russia · AS49352". Geo/ASN also render from a local MaxMind DB when configured.
- **`--malware` rebuilt into a real malware-triage tool.** It was a traffic-stats dump that reported zero downloads/C2/exfil even on textbook infections. It now correlates the file-carving, DNS, HTTP, IP-intel and SMB analyzers and leads with a **Malware Verdict** (MALWARE PRESENT / LIKELY / SUSPICIOUS / NO STRONG SIGNAL + score/reasons), then categorized evidence: **Executable Payload Downloads** (type + SHA-256 for VT pivot, with content-type-masquerade flagging), **Exploit-Kit / drive-by delivery URIs**, **C2 / beacon signals** (HTTP check-ins + periodic flows), **malicious/suspicious DNS** (DGA/tunnel/abuse-TLD/NXDOMAIN), **threat-intel matches**, **SMB lateral-movement/worm spread**, plus deterministic malware checks and triage guidance. Validated: the 2014-11-08 Nuclear-EK capture → MALWARE PRESENT (3 EXE payloads w/ hashes + EK gate URIs); Hancitor → LIKELY MALWARE (C2 beacon); 15+ benign protocol captures → NO STRONG SIGNAL (no false positives).

### Added
- **`--prconos` (ProConOS), `--csp` (Allen-Bradley CSP), `--iccp` (ICCP/TASE.2) and `--honeywell` upgraded from presence-only stubs to real analyzers.** Developed from authoritative protocol references (Redpoint nmap probes, the PCCC spec, IEC 60870-6-503 TASE.2 object names) and validated with spec-accurate captures:
  - **ProConOS** — decodes the marker/service and flags the read-project request that discloses the ladder-logic runtime and PROJECT SOURCE CODE → HIGH, **T0845 (Program Upload) / T0888**.
  - **CSP** — locates the embedded PCCC command (CMD `0x0F` + function code) and flags PLC-data writes (FNC 0x67/0xAA/…) → HIGH **T0836** and Set-CPU-Mode → HIGH **T0858**; reads and port-scans are not flagged.
  - **ICCP/TASE.2** — distinguishes ICCP from plain MMS via TASE.2 object-name markers (Bilateral_Table_ID/Transfer_Set/…) and flags TASE.2 device-control writes → HIGH **T0855/T0831** (a control command on a bilaterally-agreed point between control centers).
  - **Honeywell** — emits an asset-identification finding (T0888) when Honeywell DCS/Experion traffic is seen (the protocol is proprietary, so no command-level decode).
- **`--pcworx` (Phoenix Contact PCWorx) now identifies operations and flags PLC information disclosure.** It was a presence-only stub; it now decodes the PCWorx message type (request/response + function: connect, setup, read-PLC-info) and raises a MEDIUM detection mapped to **T0888 (Remote System Information Discovery)** when a Read-PLC-Info request (type/firmware fingerprinting — the Redpoint `pcworx-info` probe) is observed. Validated against real PCWorx traffic; no false positives on other protocols.
- **`--teamviewer` now flags remote-access-software presence (T1219).** Any TeamViewer traffic now raises an informational finding mapped to ATT&CK T1219 (Remote Access Software) prompting the analyst to confirm it is sanctioned — unapproved remote-access tools are a common exfiltration / hands-on-keyboard vector, especially on OT networks. Gated on actual TeamViewer traffic (no false positives on other protocols).
- **`--ntp` now detects mode-7 (private/monlist) amplification traffic** — the NTP reflection-DDoS vector (CVE-2013-5211) that was missing (only NTP control mode 6 was flagged).
- **`--hart` (HART-IP) now parses commands and detects write/control operations.** It was a presence-only stub; it now decodes the HART-IP PDU (message-id 3) down to the HART command number and flags the high-impact operations: **Enter/Exit Fixed Current Mode** (forces the 4-20mA loop output, overriding the live process value) → HIGH / T0831, **Device Reset** → HIGH / T0816, **Lock Device** → MEDIUM / T0878, and configuration/variable writes → MEDIUM / T0836. Validated against `hart_ip.pcap` (read commands parse to their correct HART numbers — Read Unique ID/PV/Loop Current/Dynamic Vars — and stay unflagged) plus a synthetic Fixed-Current command (→ HIGH / T0831).
- **`--fins` (Omron FINS) now detects CPU control, program-area and clear operations.** The analyzer only knew two commands (Memory Area Read/Write) and flagged just memory writes (MEDIUM), so the high-impact FINS operations were invisible. It now decodes the full command set and detects: **CPU STOP** (operating-mode → PROGRAM, halts logic) → HIGH / T0816, **RUN** (mode change) → HIGH / T0858, **Program Area read/write/clear** → HIGH / T0843, **Parameter/Program Clear** (destructive) → HIGH / T0809, and memory/parameter writes → MEDIUM / T0836, each with the specific command as evidence. Validated against `omron-FINS.pcap` (command codes cross-checked with tshark).
- **`--iec104` control-command findings now carry the specific command as evidence.** The analyzer already parsed the command type, the select/execute qualifier, the on/off state and the information-object address (IOA), but the detection only reported a generic "Control/Setpoint ASDU observed". Each finding now surfaces the actual command — e.g. `C_SC EXECUTE ON; IOA 4500`, `C_DC EXECUTE OFF; IOA 4600`, `C_SE Setpoint -43.500; IOA 5020` — flags the **EXECUTE** qualifier (the real actuation, the Industroyer breaker-trip vector, vs a select), and maps explicitly to **T0855 (Unauthorized Command Message) / T0831 (Manipulation of Control)**.
- **`--sv` (IEC 61850 Sampled Values) now detects publisher spoofing / SV injection.** A Sampled Values stream (svID) has exactly one legitimate merging unit, so the same `svID` published from two or more source MACs indicates injected/spoofed SV frames on the protection bus — a HIGH detection mapping to **T0856 (Spoof Reporting Message)**, mirroring the existing GOOSE publisher-spoofing check. The smpCnt-regression (replay) detection was already present. (`--goose` was reviewed and confirmed already comprehensive — publisher spoofing, stNum decrease/jump injection, sqNum reset, simulation/test-bit injection, ndsCom — with no false positives on a benign substation capture.)
- **`--bacnet` control operations now map to ATT&CK for ICS.** BACnet's high-consequence services were detected (HIGH severity) but unmapped: DeviceCommunicationControl (disables a device's comms) now maps to **T0814 (Denial of Service)** and ReinitializeDevice to **T0816 (Device Restart/Shutdown)**, alongside the existing WriteProperty → T0836 and object create/delete → T0809. BACnet service parsing and severities were verified accurate (WriteProperty kept at MEDIUM as the dominant normal BMS operation; DeviceCommunicationControl/ReinitializeDevice/CreateObject/DeleteObject at HIGH).
- **`--modbus` now distinguishes coil forcing from register writes for ATT&CK.** A coil write (FC 5/15) forces a digital output on/off — direct manipulation of a physical actuator (valve/relay/motor) — and is now reported as "Modbus Coil Force (output manipulation)" mapping to **T0831 (Manipulation of Control)**, while register writes (FC 6/16/22/23) remain "Modbus Write Operation" → **T0836 (Modify Parameter)**. A write-heavy capture now shows both T0831 and T0836; read-only traffic flags neither. (Modbus function-code parsing and severities were verified accurate; exception-flood and verdict fixes were already in place.)
- **OT protocol ATT&CK-for-ICS mapping now covers program transfer and CPU state changes.** The shared OT verdict mapped generic finding titles to ATT&CK-for-ICS techniques by keyword, but the S7comm titles "Program Transfer" and "CPU State Change" matched none — so the most critical Stuxnet-class operations (ladder-logic download, PLC stop/start) were left unmapped while only writes showed T0836. Added keyword mappings so a program transfer maps to **T0843 (Program Download)** and a CPU state change to **T0816 (Device Restart/Shutdown)**. An S7 capture now reports T0816 + T0843 + T0836 (+ discovery/exposure); benefits every OT protocol that uses the shared verdict (modbus/enip/cip/dnp3/goose/…).
- **`--snmp` now surfaces SNMPv3 USM usernames.** The v3/USM parser already recovered `msgUserName` (cleartext even in authPriv mode) but it was dropped; a new **SNMPv3 USM Users** section now lists the device-management accounts — a useful identity artifact for who manages the SNMP devices. Validated on a v3/USM capture (`pippo`, `pippo2`, …); v1/v2c captures omit the section, and default-community (`public`/`private`) and SET-write detections are unchanged.
- **`--winrm` now detects remote command/shell execution (WinRM lateral movement, T1021.006).** WSMan carries the operation in the WS-Addressing `<a:Action>` element inside the SOAP body, but the analyzer only matched the HTTP `SOAPAction:` header (which WSMan does not use), so remote-execution actions were missed. It now also extracts the body action URIs (`shell/Create`, `shell/Command`, `transfer/Create`, `Enumerate`, …) and raises a HIGH detection when a WSMan shell create/command is seen in cleartext WinRM. Validated: a cleartext WSMan `shell/Command` → HIGH "WinRM remote command/shell execution (WSMan) [T1021.006]"; default (encrypted) PSRemoting stays silent — an honest limitation since the message body is encrypted and commands are not recoverable.
- **`--creds` coverage expanded to the PCredz/NetworkMiner protocol set, with crackable-hash output.** New extractors: **LDAP simple bind** (cleartext bind DN + password — an immediate AD compromise), **SNMP v1/v2c community strings** (cleartext device credentials, with default `public`/`private` flagged), and **NetNTLMv1/v2 crackable hashes** built by correlating the NTLM Type-2 server challenge with the Type-3 response across the flow. A new **Crackable Hashes (offline)** section lists each captured hash with its account, type, and **hashcat mode** (`-m 5500`/`-m 5600`) ready to crack, and a `CAPTURED AUTH HASHES (offline-crackable)` verdict tier and `Crackable Authentication Hash Captured` deterministic check were added. Captured hashes are kept out of the cleartext "Discovered Secrets" table and are no longer mislabeled as "cleartext" exposure. Validated: an NTLM capture yields `Administrator`/`chewbacca` NetNTLMv2 hashes (hashcat `-m 5600`); SNMP captures surface the community string (`public`); LDAP simple-bind parsing verified against a crafted bind; FTP/benign/OT captures unaffected.
- **`--hostdetails` now has an "Observed Usernames" section.** It aggregates every account attributed to the host across all identity-bearing protocols — NetBIOS `<03>` logged-on user, Kerberos client principal (cname), NTLMSSP (binary + HTTP), HTTP Basic/form credentials, FTP `USER`, SMTP/POP3/IMAP `AUTH`, Telnet, and SMB — by folding in the credential analyzer. Each row shows the username, domain (split from `DOMAIN\user` / `user@realm`), the consolidated **source method(s)** it was recovered from, and the traffic direction (account *in use on* the host as a client vs. *authenticating to* the host as a server). The section is omitted when no accounts are observed. Validated: an NTLM/Kerberos capture surfaces `Administrator`/`chewbacca`; an FTP capture surfaces the `USER` account; an SMTP capture surfaces the `AUTH LOGIN` account.
- **`--wmic` now detects WMI remote process execution (wmiexec lateral movement, T1047).** `Win32_Process` was treated as benign "process inventory", so wmiexec-style remote command execution was missed. It now raises a dedicated HIGH detection when `Win32_ProcessStartup` appears (the strong discriminator — it carries the `Win32_Process.Create` startup info and is virtually never part of an inventory read) or `Win32_Process` co-occurs with a command line, and surfaces the actual command(s) as evidence. The command patterns were broadened to catch the Impacket signature `cmd /Q /c <command> > C:\__<file> 2>&1` and the bare `cmd /c` form (the old pattern required `cmd.exe`). Validated: the DCERPC capture → HIGH "WMI remote process execution (Win32_Process.Create) [T1047]" with the real `cmd /Q /c dir|ipconfig …` commands; a benign `SELECT * FROM Win32_Process` inventory query stays "process inventory" with no T1047. (Note: when the WMI object is marshaled in DCOM/WBEM binary form the command line is not string-recoverable from the wire — an honest dissector limitation.)
- **`--rpc` now names the DCE/RPC operations and covers coerced-auth vectors.** Operation numbers are resolved to their MS-protocol names for the abused interfaces, so the analyst sees the actual tradecraft — `SVCCTL::RCreateServiceW`/`RStartServiceW` (PsExec), `TSCH::SchRpcRegisterTask`/`SchRpcRun`, `DRSUAPI::IDL_DRSGetNCChanges` (DCSync), `SAMR::SamrEnumerateUsersInDomain` (enumeration), `ATSVC::NetrJobAdd` — instead of a bare `opnum 12`. **EFSRPC (PetitPotam, T1187)** was added as a high-risk interface (both UUIDs; rare benignly, strong coercion signal), and **MS-RPRN print-spooler** is now named with its PrinterBug coercion opnums surfaced (`RpcRemoteFindFirstPrinterChangeNotification`) without flagging benign printing. All opnum mappings were verified against the Microsoft MS-SCMR/MS-TSCH/MS-DRSR/MS-SAMR specs to avoid false attribution.
- **`--creds` now also reports interactive Telnet credentials.** It folds in `--telnet`'s reassembled-transcript results, so character-mode telnet logins (which the per-packet credential scan cannot see) now appear in the credential-focused view too — e.g. telnet `fake`/`user` surfaces as a Telnet Login pair with a CLEARTEXT CREDENTIALS EXPOSED verdict.
- **The remaining informational protocol analyzers (`--vlan`, `--qos`, `--sizes`, `--ptp`, `--aim`, `--lldp`) now carry the triage Analyst Verdict** for consistency with the rest of the tool (silent on benign captures; fires only on warning+ findings).
- **OT/ICS and several IT analyzers now lead with a triage Analyst Verdict.** The dedicated OT renderers (Modbus/EtherNet-IP/CIP/S7/GOOSE/Sampled-Values/Synchrophasor/IEC-101-103/PROFINET) already had one, but the shared `_render_industrial_summary` path used by BACnet, OPC UA, IEC-61850 MMS, CoAP, HART-IP, FINS, MQTT, Modicon, Yokogawa, Niagara Fox, MELSEC, CODESYS, ICCP, PCCC, GE SRTP, EtherCAT and Crimson did not — they now show the same "Analyst Verdict + Focus Here First" block (silent on benign captures, fires only on warning+ anomalies). The IT analyzers `--powershell`, `--wmic`, `--ssdp`, `--wlan`, `--routing`, `--nfs` and `--smtp` likewise gained the triage verdict lead. Validated: a BACnet capture → "REVIEW — BACnet: foreign device registration"; benign captures stay silent.
- **`--obfuscation` now leads with an Analyst Verdict and renders its detections.** The analyzer computed graded detections (covert-channel/exfil share, high-entropy on cleartext ports, recovered IOCs, ATT&CK-aligned attack tradecraft, base64/hex blobs) but the renderer dropped them and showed only a wall of byte/session statistics. It now opens with a verdict (COVERT CHANNEL / ENCODED ATTACK CONTENT / OBFUSCATED CONTENT OF INTEREST / ENCODED CONTENT PRESENT — likely benign / none) and a **Detections** section with severity, details and evidence. Base64/hex blobs are intentionally low-severity and do not escalate the verdict (TLS, cookies and images encode constantly). Validated: Hancitor → COVERT CHANNEL / ENCODED ATTACK CONTENT (T1071.001 tradecraft + 33.9% suspicious-byte covert-channel share); benign HTTP → NO OBFUSCATION / TUNNELING SIGNALS.
- **`--strings` and `--email` now lead with an Analyst Verdict and surface their computed checks.** Both analyzers computed deterministic checks (and `--strings` also threat hypotheses / OT / CTF markers) that the renderers dropped. `--strings` now opens with a verdict (MALICIOUS/SENSITIVE STRINGS PRESENT / OT CONTROL-WRITE / CTF MARKERS / SUSPICIOUS / none) and adds Threat Hypotheses, OT/ICS Markers, CTF Markers, and a **Deterministic String Security Checks** section. `--email` now opens with a verdict (MALICIOUS EMAIL ACTIVITY / CLEARTEXT MAIL CREDENTIALS EXPOSED / SUSPICIOUS / none), surfaces a **Cleartext Credential Exposure** section (recovered mail usernames/passwords), and adds a **Deterministic Email Security Checks** section.
- **`--secrets` now shows decoded findings by default and leads with an Analyst Verdict.** The reversible-secret decoder (Base64/Hex/URL-encoding/JWT/compressed) computed a rich set of signals — 10 deterministic checks (decoded credentials, private-key/crypto material, token/session material, encoded command/LOLBin execution, C2/exfil markers, secrets-to-public-internet, multi-stage decoding chains, high-reuse staging, OT/ICS context, CTF flag markers), threat hypotheses, and OT/CTF marker lists — but the renderer dropped all of it and hid the actual decoded cleartext behind `-v`, so the default view was just counters. `--secrets` now opens with an **Analyst Verdict** (SENSITIVE MATERIAL DECODED — INVESTIGATE / ENCODED SECRET MATERIAL OF INTEREST / CTF FLAG MARKERS / REVERSIBLE ENCODED CONTENT FOUND / none) plus reasons, shows a **Decoded Findings** table by default (top 15; `-v` for all — this is the deliverable), and adds **Threat Hypotheses**, **OT/ICS Markers**, **CTF Flag/Challenge Markers**, and a **Deterministic Secret Security Checks** section with evidence. Validated: synthetic capture with base64 credentials + PowerShell/C2 markers to a public IP → SENSITIVE MATERIAL DECODED (5 checks fire); benign HTTP → NO REVERSIBLE SECRETS DETECTED (no false escalation).
- **`--creds` now surfaces its computed IR intelligence and leads with an Analyst Verdict.** The analyzer already computed credential-attack telemetry that the renderer silently dropped — it only printed raw username/secret/pair tables. `--creds` now opens with an **Analyst Verdict** (CREDENTIAL COMPROMISE — IMMEDIATE ACTION / ACTIVE CREDENTIAL ATTACK / CREDENTIAL REUSE / CLEARTEXT CREDENTIALS EXPOSED / none) plus reasons, and adds dedicated evidence sections: **Credentials Sent to Public Internet** (cleartext creds to a routable destination — rotate), **Privileged Account Exposure** (root/admin/sa in cleartext), **Suspicious Authentication Activity** (brute-force/spray/stuffing with attempts/accounts/passwords/targets), **Credential Reuse Across Hosts** (same secret to 3+ destinations → lateral movement), **Token Reuse / Fan-out**, a **Hit Confidence** breakdown, and a **Deterministic Credential Security Checks** section. Validated on `common-protocols/ftp/bruteforce.pcap` → ACTIVE CREDENTIAL ATTACK (60 attempts vs account `bro`, 30 distinct passwords); clean OT/benign captures → NO CLEARTEXT CREDENTIALS DETECTED (no false positives).
- **`-h`/`--help` output is now colorized and includes an EXAMPLES section.** Group headings, the usage line, flags, and metavars are colored, and a new EXAMPLES epilog shows common invocations (`--overview`, `--malware`, `--ips --ip-geo`, etc.). On Python 3.14+ the help defers to argparse's native color theme (no double-coloring); on older Pythons a built-in fallback pass colorizes the same elements. Color is suppressed for `--no-color`/`--quiet`, when `NO_COLOR` is set, and when output is piped (any ANSI argparse emits natively is stripped in those cases so the flags are always honored).
- **Per-function help (e.g. `pcapper --tcp -h`, `pcapper --files -h`) is now colorized too.** The "PCAPPER FUNCTION HELP MODE" view colors its title, the `Function:`/`Description:`/`Usage:`/`Related options:` labels, the function flag, the usage metavars, and each related-option flag, with the related-options column kept aligned (padding is computed on the uncolored text so the zero-width ANSI doesn't shift it). Honors the same `--no-color`/`--quiet`/`NO_COLOR`/non-TTY suppression as the global help.
- **`--ldap`, `--ntlm`, `--smb`, `--rpc`, `--netbios`, `--winrm`, `--wmic` now show a "Deterministic … Security Checks" section.** These analyzers each computed 7–9 deterministic checks with evidence (e.g. cleartext LDAP exposure, NTLMv1 legacy auth, SMBv1/unsigned-SMB, admin-share lateral movement, SAMR enumeration, WMI remote-execution tradecraft) that previously only fed the `-summarize` threats rollup and were never visible in the analyzer's own output. A shared renderer now surfaces them with per-finding evidence (label + evidence, cleared checks collapsed).
- **`-view` now works with `--streams`** to dump the reassembled HEX/ASCII content of the displayed streams (bare `-view`; combine with `-ip`/`-port`/`-search`/`-id` to scope). It retains up to 64 KB per direction for the view and caps at 10 streams unless `-v` is given. The `-view FILENAME` behavior for `--files` is unchanged.
- **`--hostdetails` now shows a Host Activity Timeline** — the per-host chronological record (connections, DNS, NetBIOS, file transfers, etc.) that was already collected internally but never rendered. This is the core forensic/IR artifact for a single host (top 40 events, or all under `-v`).
- **`--hostdetails` surfaces the host's HTTP User-Agent(s)** in Host Identity and feeds them into OS/device inference — the strongest passive OS-build/client/tooling signal (e.g. a `Windows NT 6.1` UA now correctly classifies the host as Windows and shows the exact client).

### Changed
- **`--hostdetails` dropped a redundant full-capture pass** (`analyze_pcap`), reusing the already-computed IP summary for packet totals and the capture window. Note the fast path remains chaining `--threats --hostdetails` (the cross-protocol threat correlation is then computed once and reused).

### Fixed
- **`--vnc` now parses RFB security types from the binary handshake instead of matching text.** The auth-type detection matched the words "None"/"Tight"/"Ultra" as substrings anywhere in the payload, so the ubiquitous word "none" in framebuffer/clipboard data produced a constant false "VNC sessions with no authentication detected" alert (it fired on a capture that actually used VNC Authentication). It now parses the RFB ProtocolVersion + security-type message (handling RFB 3.3's 4-byte form and 3.7/3.8's count+list form) and reports the real negotiated type. A server offering **type 1 (None)** is now correctly HIGH ("grants remote control with no password", T1021.005); **type 2 (VNC Authentication)** is flagged as weak DES challenge/response (offline-crackable, 8-char password). Validated: a real capture using VNC-auth/Apple-RD no longer false-alarms; a synthetic None-auth handshake fires HIGH.
- **`--cip`/`--enip` PLC Start/Stop commands are now HIGH severity.** A CIP Stop (service 0x07) halts the PLC and a Start (0x06) changes its operating mode — denial-of-control / mode-change attacks — but they were rated MEDIUM while only program-download/reset were HIGH. They are now HIGH, consistent with S7comm PLCStop and DNP3 restart (both HIGH/CRITICAL) and mapped to T0858 (Change Operating Mode). Plain tag/attribute writes remain MEDIUM (routine on a live cell); read-only CIP traffic is unaffected. (CIP service-code parsing — including the earlier 0x4C/0x4D read/write correction — and the per-service ATT&CK map were verified accurate; `--dnp3` was reviewed and confirmed already excellent: control commands → HIGH with explicit T0855/T0831, restarts → T0816, writes → T0836.)
- **`--ldap` "invalid credentials" detection no longer lists benign result codes.** The detail string echoed *every* observed LDAP result code, so a normal `14 (saslBindInProgress)` SASL continuation appeared alongside the real `49 (invalidCredentials)` as if it were a failure. It now lists only the failure codes (excluding success/compare/saslBindInProgress); `14` still appears in the Response Codes table where it belongs. Benign AD captures show no false invalid-credential finding.
- **`--hostdetails` "Observed Usernames" section now always renders.** It was omitted entirely when no accounts were recovered, which read as the feature being broken. It now always appears: when no authenticated protocol carried a username it states so explicitly and points to the host/computer name that *is* available (e.g. a capture exposing only the NetBIOS workstation name `WIN-…`, not a user account). Hosts that do carry credentials still show the full username table.
- **`--netbios` no longer false-flags normal name resolution as NBNS spoofing.** It counted "multiple hosts *queried* the same name" as a name conflict, so benign shared lookups (WORKGROUP, WPAD, a common server) produced a flood of bogus HIGH conflicts (100 on a WannaCry capture) — querying a name is not claiming it. That check was removed; the real signal (a *unique* name *answered* by multiple owners — Responder-style poisoning) is kept and hardened to exclude group registrations that legitimately have many responders: the NBNS group bit, group suffixes `<1C>/<1D>/<1E>`, and well-known group names (WORKGROUP/MSHOME/MSBROWSE). The verdict ribbon also no longer shows empty `-` titles for anomaly findings (it falls back to the anomaly category). WannaCry NBNS now reports 0 conflicts (correct — its attack is SMB/EternalBlue), while a genuine unique-name poisoning still fires.
- **`--powershell` now detects the short `-e`/`-ec` EncodedCommand forms.** The encoded-command *detection* only matched `-enc`/`-encodedcommand`/`frombase64string`, so the very common `powershell -e <base64>` cradle was missed even though the decoder already handled `-e`. The detection now matches any unambiguous EncodedCommand prefix (`-e`/`-ec`/`-enc`/`-encodedcommand`) when followed by a base64 blob (so a benign `-e` flag value can't trip it) and aligns with the decoder. Verified end-to-end: a `powershell -e <base64>` payload now fires HIGH and the decoded cleartext (`IEX (New-Object Net.WebClient).DownloadString(...)`) is shown as evidence; benign PS-remoting and `-ExecutionPolicy` stay silent.
- **"Observed Plaintext" sections no longer dump binary noise.** The protocol string-carver emits every printable 4+ byte run found in binary payloads, so the verbose carved-string view (shared by `--rpc`, `--telnet`, `--ssh`, `--smtp`, `--snmp`, `--netbios`, `--nfs`, `--wmic`, `--winrm`, and others — 12 renderers) was flooded with junk like `@gE::` and `Cj::#6?\`. A shared meaningfulness filter now keeps only entries that contain a real word and are predominantly text/path characters, so the section shows genuine forensic artifacts (hostnames, UNC paths, `\pipe\lsass`, WMI `Win32_Process`/`CommandLine` indicators) instead of noise.
- **`--telnet` now recovers credentials from interactive (character-mode) sessions.** Username/password extraction ran the regexes against each packet individually, but character-mode telnet sends one keystroke per packet with the server echoing it back, so a `login:`/`Password:` prompt and its typed value never appear together in any single packet — interactive logins were missed entirely. The analyzer now reassembles a chronological per-session transcript (Wireshark Follow-Stream style) and runs a second credential pass over it, collapsing the echo-doubling (`ffaakkee` → `fake`). It also no longer mistakes the `Last login: <weekday>` MOTD banner for a login prompt. Validated on both character-mode and line-mode telnet captures (recovers `fake` / `user`).
- **`--secrets` hex decoder no longer reports random binary as secrets.** A random 8-byte value (e.g. `3001805773435453`) decodes to ~6/8 printable bytes by chance and, after the non-printable bytes were stripped for display, looked like a short word (`0WsCTS`) that passed the meaningfulness gate. The hex path now requires a high raw printable ratio (≥90%, evaluated before stripping) — genuine hex-encoded credentials/tokens are essentially fully printable — eliminating the false hits while base64/JWT keep the looser gate. A benign HTTP capture that previously surfaced bogus hex "secrets" now reports none.
- **`--modbus` no longer floods on exception-heavy captures.** Modbus exception findings were aggregated by `(exception, function-code, src, dst)`, so a capture probing many function codes against one device produced dozens of near-identical findings (87 on a test capture) that buried the real signal. Exceptions are now rolled up by `(exception-type, src, dst)` with the function breakdown folded into the description — e.g. `Exception 1 (Illegal Function) across 104 function code(s) (x104)`, the actual reconnaissance/fuzzing signal. The test capture dropped from 87 findings to 11. (Completes the OT per-packet-flood pass alongside the earlier dnp3/enip/cip fixes; s7's diverse write/program-transfer findings were verified to be genuinely distinct, not a flood.)
- **`--strings` no longer cries "MALICIOUS" on benign web traffic.** The LOLBin check matched a bare `cmd`/`powershell` as a word, so a CSS class like `.btest-cmd` tripped "command execution"; it now requires real execution context (`.exe`, a command-line switch, `IEX`/`Invoke-Expression`, `cmd /c`, `certutil -…`, etc.). The cleartext-credentials check fired on any `username:` field name (ubiquitous in JS/JSON/HTML); it now requires a plausible password *value* (rejecting JS scaffolding like `[]`, `{}`, functions and template placeholders). Benign `bro.org` dropped from MALICIOUS to a mild SUSPICIOUS; Hancitor malspam still flags correctly.
- **`--email` "Messages" count is now accurate.** The counter was incremented once per parsed protocol line, so a single 78-packet message reported "1291 Messages" (and benign web traffic showed dozens). It is now derived from real message boundaries — unique Message-IDs, else SMTP DATA transactions, else POP3 RETR / IMAP FETCH retrievals (sample IMF capture: 1291 → 1; benign HTTP: 62 → 0).
- **`--email` no longer misclassifies web browsing as email or flags `.js` URLs as attachments.** A plain software-download path (`/download/foo.tar.gz`) matched the generic "Attachment" webmail path-keyword, so any site with a `/download` URL was treated as webmail and its page mined for "attachments" (`/statuses.js`, `…/user_timeline.js`) → a bogus "MALICIOUS EMAIL ACTIVITY / dangerous attachments" verdict on benign HTTP. Webmail classification now requires a real webmail host/referer/User-Agent or a webmail-specific path (not generic `/download`,`/login`,`/api/`), and loose attachment-name extraction (`name="…"`, bare `*.js`/`*.txt`/`*.tar.gz`) only runs on actual MIME part headers (Content-Disposition/Content-Type/filename). Also drops quoted-printable artifacts (`=3D` → `3D`) from recovered mail passwords. Benign `bro.org` → NO HIGH-RISK SIGNAL; real SMTP still recovers credentials and fires its verdict.
- **`--creds` brute-force detection now catches single-account password guessing.** The auth-abuse classifier only fired on password spraying (≥4 distinct accounts) or stuffing (≥4 destinations), so the most common pattern — one account, one target, many passwords (e.g. an FTP/SSH guessing run) — was missed entirely. It now also classifies **password brute-force** by distinct-secret fan-out per (account, target), and labels each detected source with its pattern(s). The local public-IP test was also switched to the shared `is_public_ip` helper so IPv6 multicast is no longer counted as a "credential sent to the public internet".
- **`--ips` port-scan detection no longer false-positives on servers, and now catches vertical scans.** Scan profiling counted the destination port of *every* packet, so a DNS/HTTP server responding to clients (service src port → many client ephemeral dports) was flagged as a "high-port sweep". It now only counts client-initiated flows (ephemeral source port). Separately, a single-target multi-port scan (the most common nmap scan, e.g. synscan.pcapng's 1000-port sweep of one host) was missed because both profile types required ≥3 destinations — added a `vertical_scan` profile.
- **`is_public_ip` no longer misclassifies IPv6 multicast as a public peer.** It tested only `is_global`, which Python reports `True` for global-scope IPv6 multicast (and `ff02::` solicited-node groups), so benign IPv6 Neighbor Discovery / mDNS could be flagged as public-internet exposure across every analyzer. It now also requires non-multicast.
- **Deterministic-checks audit (ICMP, IPS, TCP, UDP, protocols, services, FTP, files, timeline, hosts, domain, CTF):** systematically reconciled every "Deterministic … Security Checks" section so the categories shown match what the analyzer actually computes. Removed phantom labels that always rendered "none" (ICMP, IPS, TCP, UDP, protocols, services, files, hosts). Surfaced **dropped** checks that were computed-with-evidence but never rendered: **domain** now shows 10 additional AD-attack checks (kerberoasting/weak-cipher, AD CS/certificate abuse, DCSync fingerprint, LDAP directory abuse, relay/coercion chain, credential material in domain flows, identity attack-path, public exposure, baseline deviation, OT identity overlap); **timeline** adds Remote Execution Tooling + Cleartext Remote Admin; **CTF** adds credential/passphrase checks. **FTP** gained real Cleartext Credential Exposure and Anonymous/Guest Abuse checks (wired from existing detections), and **ICMP** gained `icmp_zone_boundary_exposure` (internal↔public ICMP) with evidence.
- **Deterministic-checks audit (DNS, ARP, exfil):** the `--dns` "Deterministic DNS Security Checks" section now also renders the `opcode_or_any_abuse` and `dns_beaconing_periodicity` checks (computed but previously dropped), and abuse-prone TLD volume now feeds the DGA check with evidence. The `--arp` section now renders the analyzer's own computed checks (13 categories with per-finding evidence) instead of re-deriving 8 via fragile free-text keyword matching. The `--exfil` section gained a "Deterministic Exfiltration Checks" block surfacing the analyzer's computed checks (DNS tunnel, HTTP POST exfil, OT control-channel exfil, staging→exfil, file exfil, etc.) that previously only fed the threats rollup.
- **`--hostdetails` now reports the host MAC address even without ARP/DHCP.** MAC extraction only consulted ARP conversations and DHCP sessions, so a capture with no ARP (e.g. a mid-session HTTP capture) showed "MAC Address: -" despite every Ethernet frame carrying the address. It now also uses the per-IP MACs the IP pass already records from frame headers, and filters out broadcast/multicast addresses (e.g. `ff:ff:ff:ff:ff:ff`) that are destinations seen in the host's traffic, not its own identity.
- **`--streams` reassembly no longer injects Ethernet-padding null bytes.** Pure-ACK/control segments in sub-60-byte frames carry a Scapy `Padding` layer; the fallback `bytes(tcp.payload)` pulled those nulls into the reassembled stream (e.g. a request showed `\x00\x00\x00\x00\x00\x00HTTP/1.1` instead of `GET / HTTP/1.1`). Segments without a `Raw` layer are now treated as empty, fixing the stream content view, `-id` follow, and stream search matching.
- **`--hostdetails -ip` "Remote Services Established By Host" no longer reverses client/server roles.** The section keyed off raw IP-header direction, so a host that only *received* inbound connections (e.g. a web server) had its own response traffic mislabeled as outbound — it was listed as "establishing remote services" to its clients, on its own service port (80) plus the clients' ephemeral source ports (49xxx). It now uses the handshake-confirmed service ports (who actually listens) to identify the *peer's* service the host connected to: a pure server correctly shows "No remote services established," while a client shows only the real remote service ports it reached (its server role still appears under "Services Hosted On This Host").
- **`--files` no longer reports bogus X.509 certificates on ASN.1/BER protocols.** The raw DER carver keyed only on the ASN.1 SEQUENCE tag (0x30), which every BER protocol shares, so MMS (port 102), LDAP, SNMP, Kerberos, etc. had their PDUs carved as fake "X509AF" certificates — an IEC-61850 MMS capture with zero TLS reported 31 certs. DER blobs are now validated as actual certificates (structural gate + parse via the cryptography library); counts now match ground truth (a TLS capture went from 3720 bogus → 3 real certs, exactly tshark's unique-serial count).
- **`--files` now carves HTTP/stream file transfers again (default, non-`-ip` runs).** When reusing pre-loaded Scapy packets, the carver read TCP payloads via `bytes(tcp.payload)`, which pulled in the Ethernet frame-padding bytes Scapy attaches to short segments (e.g. pure ACKs). Those null bytes were injected into the reassembled TCP stream, corrupting it (an HTTP response's `HTTP/1.` became nulls) so the body was mis-detected as HTTP/2 and dropped — a capture full of HTTP downloads reported zero files. Payload extraction now excludes the `Padding` layer, matching the dpkt/disk path byte-for-byte (a 250-artifact malware capture went 0 → 250 files; the `-ip` path already used the disk parser and was unaffected).
- **`--hostnames -ip <addr>` no longer mis-attributes a host's traffic as its identity.** The previous "related" scoping force-mapped every hostname seen on any packet to/from the target onto the target IP, so a client that merely *queried* or *connected to* `google.com`, `microsoft.com`, etc. was wrongly listed as owning those names in the IP ↔ Hostname Correlation. Scoping now keeps only findings whose session-aware mapping already points at the target — a DNS A-record resolving to it, a Host header/SNI where it is the server, or its own NetBIOS/DHCP/mDNS announcement — so `-ip` shows the host's real identity (e.g. its `WIN-…` NetBIOS name) and a server IP correctly shows the name(s) that resolve to it. The per-method, direction-aware extraction (HTTP Host/SNI → server, NTLM Type3 → client, SMTP HELO → client, certificates → server, DNS query names → unmapped, etc.) was already correct; this removes the downstream remap that was discarding it.

## 2.0.0 — 2026-06-15

**v2.0.0 — the threat-hunt / incident-response release.** This major version turns pcapper from a protocol-analysis tool into a threat-hunting, forensics, and IR/triage platform for IT *and* OT/ICS networks. Across this cycle every analyzer was reviewed function-by-function against that vision: each now renders an **Analyst Verdict** with severity, **evidence on every detection**, and **ATT&CK / ATT&CK-for-ICS** mapping; ~15 new OT/ICS protocol analyzers and major OT correctness fixes landed; a new **IT→OT pivot** detector and **JA3/JA4**, **cert-IOC**, **DoH-over-SNI**, and **Net-NTLM-hash** hunting were added; and a sweeping correctness / false-positive audit plus a performance + de-duplication pass rounded it out. The detailed per-feature entries from this cycle are listed below.

### Added (2.0.0 headline features)
- **Inbound remote-access ("Remote IN") + IT→OT pivot detection (`--timeline`, `--threats`, `--overview`, `--compromised`)**: completed inbound remote-access sessions (SSH/Telnet/RDP/VNC/WinRM/SMB-PsExec/rlogin/X11) to the focused host appear as high-risk "Remote IN" timeline events (external source = CRITICAL, internal = HIGH). The aggregate analyzers raise a CRITICAL **"IT→OT pivot"** finding when a host that accepted remote access then issues an OT/control command to another host — the canonical industrial-intrusion sequence (ATT&CK T0859/T0855).
- **`--timeline` now includes CIP and EtherNet/IP commands** alongside Modbus/DNP3/S7/IEC-104/OPC-UA, so an engineering action shows in the host chronology right after the remote access that enabled it.
- **`--encrypted-dns` DoH-over-TLS detection via resolver SNI**: real DoH is encrypted on 443, so pcapper flags TLS sessions whose SNI matches a known public DoH resolver (Cloudflare, Google, Quad9, OpenDNS, AdGuard, NextDNS, …) — a common DNS-monitoring-bypass / C2-exfil evasion channel.
- **`--iec101_103` control-command detection**: command-direction ASDUs (setpoint/single/double/regulating/reset) carrying an Activation cause are flagged HIGH (T0855, reset → T0816) instead of being counted silently.
- **OT anomalies now carry explicit ATT&CK-for-ICS techniques and evidence** across CIP, Modicon UMAS, MMS, OPC UA, EtherCAT (FoE), and the shared OT exposure/scan/beacon/exfil detectors — accurate mapping rather than title inference.
- **OT protocol analyzers show full output by default** (no `-v` needed) and dropped the truncation footer — OT triage shows everything up front.

### Fixed (2.0.0 audit highlights)
- **Revived dead detectors**: `--rpc` admin-share/named-pipe (PsExec) attribution and `--wmic` WMI event-subscription persistence (T1546.003) were silently dead from double-escaped regexes; `--modicon` UMAS CPU start/stop rendered literal `(0x{:02x})` from a broken f-string; `--protocols` NULL-scan, `--icmp` IPv6-unreachable counting, `--rdp` decrypted-identity extraction, and the `--threats` IT→OT kill-chain correlation were all fixed.
- **False-positive reduction**: SMB guest/null-session flag constants; certificate leaf-only checks running on CA certs; HTTP webshell URI substring matches; multicast/broadcast treated as UDP servers; DNS concentration/poisoning/fast-flux truncated to one domain; per-client (not per-flow) PowerShell/WMI beaconing; pure-ACKs counted as TCP retransmissions; SSRF `localhost`-substring matches; the `user:`/`superuser` credential-indicator FP; and OpenVPN-over-TCP opcode parsing.

### Performance & internals (2.0.0)
- **Sub-analyzer memoization**: OT analyzers with their own packet loop (CIP/ENIP/Modbus/GOOSE/DNP3/SV/synchrophasor/beacons) are `@memoize_analysis`-cached, so `--overview`/`--threats`/`--timeline` no longer re-parse the capture 3-4× per OT protocol (~13%+ faster on OT captures; output byte-identical).
- **De-duplication (~2,000 lines removed, behaviour-preserving)**: a shared `_counter_table` renderer (131 sites, verified byte-identical), a shared OT public-exposure helper (23 modules), a unified `OT_PORT_PROTOCOLS` map (closing an OT host-detection coverage gap), and consolidation of duplicated IP-classification / entropy / port-set constants. Removed dead code (unused functions, a dead Modbus duplicate loop, debug prints).

### Added
- **`--dns` high-abuse-TLD query detection (malware/phishing staging)**: DNS tracked per-TLD query counts but never flagged the TLDs disproportionately abused for malware and phishing. Added a curated `_HIGH_RISK_TLDS` set (the Freenom free TLDs `.tk/.ml/.ga/.cf/.gq` plus the cheap new-gTLDs with the highest per-capita abuse scores — `.top/.xyz/.shop/.online/.xin/.click/.icu/.sbs/.buzz/…`, per Interisle's 2025 Phishing Landscape and Spamhaus domain-reputation data) and a WARNING that fires when a host resolves ≥5 *distinct* domains under those TLDs, with the per-TLD breakdown and a pointer to correlate with `--http`/`--tls`. The large general-purpose TLDs (`.com/.net/.org/.info/.cn/.ru`) are deliberately excluded — they lead phishing by raw volume but are overwhelmingly legitimate — and the distinct-domain gate prevents a single benign `.xyz` lookup from firing. Validated: benign DNS captures don't fire; the TLD classifier correctly excludes `.com/.net`.
- **`--certificates` Cobalt Strike default-certificate IOC detection**: the analyzer captured each certificate's serial and SHA-1/SHA-256 fingerprints but didn't check them against known-bad certs. Added a curated known-malicious certificate set (keyed by SHA-256 and by default serial) seeded with the Cobalt Strike default Team Server certificate (SHA-256 `87f2085c…`, serial `146473198` / `0x8bb00ee` — a definitive C2 IOC published by Elastic/abuse.ch). A match raises a prominent **"Known-Malicious Certificate IOC Match"** (CRITICAL) section with the offending flow, serial and fingerprint, and pushes the certificate verdict to high-confidence. Exact-hash/serial matching means zero false positives — verified on benign TLS captures with real Google/RDP/OpenSSL certificates.
- **`--tls` JA3 fingerprint threat-intelligence matching (TLS C2 / malware hunting)**: the analyzer computed JA3/JA4 fingerprints but never checked them against known-bad fingerprints — the primary technique for spotting malware/C2 inside encrypted traffic. Added a curated JA3 intel set (abuse.ch SSLBL malware families — Dridex, TrickBot, Qakbot, Tofsee, JBifrost — plus public C2-research fingerprints for Cobalt Strike / Meterpreter sockets) and a detection that flags matches. Confidence is tiered to control false positives: malware-family fingerprints raise a HIGH finding, while the Windows-socket beacon fingerprints (which collide with legitimate Windows TLS) are surfaced at INFO with an explicit "corroborate" caveat. Matches carry the malware label and observed SNI, flow into `--threats`/`--overview`, and produced zero false positives across the benign TLS sample captures.
- **`--files` now computes and displays file hashes by default (forensic/IR core deliverable)**: MD5/SHA-256 of every reconstructed file were only computed when `--extract`/`--view`/`--hash` was passed, so the default triage view had no hashes — the single most useful artifact for VirusTotal / threat-intel correlation and known-bad matching. The carved bytes are now always retained just long enough to hash (then released, so memory isn't held), and a new **"File Hashes (VirusTotal / threat-intel lookup)"** section lists each file's type, size, MD5 and SHA-256. Verified on Hancitor (the `eFax_marshajones8.doc` macro-doc and `tv.dat` payload now have lookup-able hashes) and benign captures.
- **`--files` file-type detection now recognizes the common malware delivery formats**: `detect_file_type_bytes` (shared magic-byte typer) missed **OLE2/Compound File** (`D0 CF 11 E0 …` — legacy Office .doc/.xls/.ppt, the dominant macro-malware container used by Hancitor/Emotet/Dridex), plus RAR, 7-Zip, Microsoft CAB, Mach-O and Windows LNK. All added (exact magic-byte prefixes, no false-positive risk). Hancitor's `eFax_*.doc` now correctly types as `OLE2/Office` instead of `BINARY`.
- **`--bsap` / `--genisys` — new SCADA protocol analyzers (presence + internet-exposure)**: two more OT protocols that had sample captures but no coverage. **BSAP-IP** (Bristol/Emerson gas/oil/water SCADA, UDP/TCP 1234/1235) is detected by port; **Genisys** (rail-signaling / substation master-slave SCADA) is detected by its unambiguous frame shape (control-byte start `0xF1/0xF2/0xFB/0xFD`, single trailing `0xF6` delimiter, small frame) rather than its ambiguous TCP port 10001 — a bare "ends in 0xF6" test false-matched large binary payloads, so the signature also requires the 0xF6 to appear only at the end and the frame to be small. Both flag public-IP exposure (HIGH) — an internet-reachable BSAP/Genisys SCADA endpoint is a serious finding. Their vendor framing has no authoritative public function map, so these are presence/exposure detectors rather than command decoders. Validated: BSAP 7,885 frames and Genisys 688 frames on the sample captures, with **zero** false positives on WannaCry/CIP (the tightened Genisys signature eliminated 4 initial false matches on SMB binary).
- **`--synchrophasor` — new IEEE C37.118 (PMU/PDC) analyzer for power-grid synchrophasor traffic**: pcapper had no coverage for C37.118, the protocol that carries Phasor Measurement Unit data used for transmission-grid monitoring and protection (NERC CIP relevant), even though sample captures were present. The new analyzer parses C37.118 frames (sync `0xAA`, frame type in the second byte) on TCP/UDP 4712/4713, breaks down frame types (Data / Config / Header / Command), tracks PMU/PDC ID codes, and decodes command frames. It flags (HIGH) a **"Turn OFF data transmission"** command as a denial-of-view against grid monitoring, surfaces all command frames (WARNING) for authorized-source confirmation, and flags public-IP exposure. Validated against the C37.118 TCP/UDP sample captures and a 14k-frame synchrophasor capture (frame types, ID codes, and turn-off commands all decode correctly), with no false positives on non-C37.118 traffic (SMB/CIP → 0 frames). Wired into the CLI (`--synchrophasor`) and reporting.
- **`--dns` hunting enhancements**: (1) **MX-enumeration detection** — endpoints rarely query MX records, so a host issuing many MX lookups is enumerating mail servers (spam-bot / mass-mailer / malspam, e.g. Hancitor, or mail-infra recon). A new warning fires at >=30 MX queries with the top client and a pointer to correlate with `--smtp`/`--email`. (2) **Case-folded domain aggregation** — DNS names are case-insensitive (RFC 4343), but the "Most Queried Domains" view previously split the same domain across 0x20 case-randomized forms (`ASPMX.L.GOOGLE.com` vs `aspmx.l.google.com`), fragmenting query volume; query names are now folded to a canonical lower case so true per-domain volume is shown.

The detection now reports the ATT&CK tactics implied and escalates to high only on a corroborated multi-marker tooling chain.
- **`--threats` DNS-tunneling entropy false positives**: an entropy-only threshold (≥3.8 over the full qname) flagged benign CDN/cloud names. It now requires a genuinely long encoded label, an oversized qname, or high entropy *combined* with a long label, and excludes reverse-DNS.
- **SMBv1 over-escalation**: SMBv1 presence is a vulnerability/hygiene finding, not active-intrusion evidence, and was emitted as `critical` from two code paths (outranking live attack activity in the verdict). It is now a single `high` finding with remediation/triage guidance.
- **`--threats` mislabeled OT recon as credential brute-force**: a high SYN count with near-zero SYN-ACK is connection probing/half-open scanning (or a dead service), not credential brute-force, which requires completed handshakes. Renamed "Potential TCP brute-force/credential probing" → "Potential TCP connection probing (unanswered SYNs)" so it classifies as reconnaissance (correctly attributing an OT host sweeping EtherNet/IP devices).
- **`--compromised` phantom campaigns and IOC noise**: the free-text IOC extractor's filename regex allowed spaces and greedily swallowed whole detail sentences as one "filename"; campaign correlation then grouped hosts on weak/benign shared tokens. The filename regex is tightened, and campaign clustering now requires a strong indicator (URL, hash, or public IP/routable domain). The previously dead `benign_automation_likely` deterministic check is now populated.

### Changed
- **`--threats` detection table now shows a meaningful Confidence column** for every row: detections without an explicit confidence field derive one from the computed likelihood (severity + hit count + peer fan-out + corroboration) instead of rendering a bare "-", aligning the output with triage workflows.

### Added
- `--hostnames`: raw TLS ClientHello SNI parser — SNI is now recovered on ANY port (RDP-on-3389 TLS, 8443, CONNECT-tunneled handshakes), not just the port-443 traffic scapy dissects.
- `--hostnames`: raw DER certificate SAN/CN extraction as a fallback when scapy does not dissect the TLS Certificate message (non-443 ports, tunnels, mid-stream records). RDP servers commonly present a self-signed certificate whose CN is the machine hostname.
- `--hostnames`: NTLM Type 2 CHALLENGE TargetInfo parsing (AV pairs 1/3) — the server's own NetBIOS and DNS computer names, the server-side counterpart to the existing Type 3 client workstation field.
- `--hostnames`: DNS/mDNS/LLMNR A/AAAA *query* names are now recorded as unmapped hostname evidence ("query name (unresolved)") — vital for one-directional sensor exports that contain no responses (one query-only hunt capture went from 0 to 243 discovered hostnames). Query names are deliberately NOT attributed to the querier's IP.

### Fixed
- **`--hostnames` mapped proxy-form HTTP requests to the proxy's IP**: for `CONNECT host:443` and absolute-URI requests, the packet's destination is a forward PROXY, yet the named host was recorded as mapped to that dst IP — a false hostname→IP attribution (e.g. `us2.*.rapid7.com` pinned to an internal proxy). Proxy-form authorities are now reported unmapped with the proxy noted in details; a Host header that disagrees with the request authority is flagged separately. CONNECT flows are tracked so SNI/certificates seen *inside* the tunnel are also not attributed to the proxy IP. Each CONNECT also no longer double-counts (previously emitted both a "Host header" and an "authority" finding for the same packet).
- **`--hostnames` session-direction fixes**: RDP mstshash cookie attributed to the side NOT on port 3389 (was unconditionally src); SSH identification strings attributed to their sender (was always the port-22 side); SMTP HELO/EHLO attributed to the true client side (was unconditionally src); Kerberos SPN hosts / LDAP dNSHostName left unmapped (they name directory objects — often third hosts — not the KDC/LDAP server on the flow); DHCP DISCOVERs with no usable address are left unmapped instead of pinned to 0.0.0.0.
- **`--hostnames` RDP mstshash mislabeled as a host token**: per MS-RDPBCGR the cookie carries the client's login identity (`username` or `DOMAIN\username`). `DOMAIN\username` values are now parsed and reported as "RDP mstshash cookie domain"; bare values are labeled as client login identity (typically the username).
- **`--hostnames` SSH banner noise**: the extractor tokenized binary KEX packets, reporting algorithm names ("3des-cbc", "ssh-ed25519") as hostnames by the thousands (4,691 junk findings in one capture). Per RFC 4253, only the identification string's comment field can carry a hostname; everything else is now ignored.
- **`--hostnames` dropped real machine names containing underscores** (e.g. SNI `nat_hydra` from an RDP server): hostname validation now accepts `_`, which is invalid DNS but common in NetBIOS-derived names. IP-literal SNI values (e.g. `SNI=10.180.81.123`) are now skipped instead of reported as hostnames.

### Added
- `--cache-mb MB` flag: raise the in-memory packet cache budget (total and per-file) for the run so large captures are parsed once and shared across chained analysis steps; `--cache-mb 0` disables caching. Env equivalents: `PCAPPER_CACHE_MAX_BYTES`, `PCAPPER_CACHE_FILE_MAX_BYTES`, `PCAPPER_CACHE_ENABLED=0`.

### Changed
- Major performance: chained analysis steps (e.g. `--ssh --tls --dns ...`) now share a single preloaded packet list via the forced-packet-view hook in `get_reader()`. Previously, analyzers without `packets=`/`meta=` parameters (≈60 of the step analyzers, plus everything nested under `--overview`, `--hostdetails`, `--files`, `--ctf`, and the industrial protocol helpers) re-read and re-parsed the pcap from disk on every step for captures above the per-file cache limit. A chained run now parses the capture exactly once.
- `get_reader()` now serves packets straight from the in-process cache when the capture is already loaded, regardless of the per-file size gate that controls *new* cache loads (fixes per-step disk re-reads for 64–256 MB captures the CLI had already preloaded).
- Single-step runs of fan-out analyzers (`--overview`, `--hostdetails`, `--files`, `--ctf`) now preload the packet list once instead of letting each nested analyzer re-read the capture (`--overview` alone previously triggered ~30 full pcap parses).
- Capture loading throttles progress-bar updates (every 512 packets instead of per packet) and hoists loop-invariant checks out of the read loop.
- Per-packet hot loops dissect transport layers once per packet instead of repeatedly: `--ips` (TCP/UDP layer reuse across protocol inference, conversation tracking, and per-endpoint accounting), `--routing` (IP/IPv6 reuse for proto/payload extraction), `--dns` (single UDP dissection per packet), `--timeline` (DNS/UDP reuse), `--threats` (single suspicious-marker scan per payload).
- BPF host-only filtering pre-parses the host address once per run instead of per packet.
- Run-level analysis memoization: analyzers that are invoked more than once within a single run (e.g. `--ips` runs the hostname analyzer internally; `--threats` fans out into ~35 sub-analyzers; `--overview`/`--hostdetails`/`--files` fan out similarly) now return a deep copy of the first result instead of re-iterating the capture. Keys include capture identity (path/size/mtime), the packet view in effect, and all analyzer arguments, so filtered and unfiltered analyses never alias. Disable with `PCAPPER_ANALYSIS_MEMO=0`.
- Layer-walk reduction in the heaviest per-packet loops (profiled: ~28M `haslayer` + 16M `getlayer` calls per 7-step run on a 144k-packet capture): `--protocols` (~25 chain walks per packet down to ~8), `--hostname` (12 extractor helpers now share one dissection per packet, ~70% fewer walks), `--icmp` (ICMPv6 detection chain resolved once), `--vlan`, and `extract_packet_endpoints()` (used by ~30 modules) now use single `getlayer` resolution.
- `packet_length()` helper: `len(pkt)` on a scapy packet re-serializes it (~14µs); hot loops now read `len(pkt.original)` (~50x faster) across 11 modules.
- `lru_cache` on per-IP classifiers that previously re-parsed IP strings per packet (`routing`, `ips`, `hostname` public/private checks; ~1M redundant `ipaddress` parses eliminated per run).

### Fixed
- Latent `TypeError` in the `-port` target-candidate resolution path: `analyze_ips()` was called with an unsupported `include_threat_hunting` keyword.
- **Key/value column misalignment with color enabled**: `_format_kv` (used by ~960 report lines) applied width padding to the already-colorized label, so the ANSI escape bytes counted toward the column width and the `: value` column drifted out of alignment whenever color was on. Padding is now computed on the visible label width (byte-identical output when color is disabled).
- **Local-time leak in threat-narrative timestamps**: the threats analyzer rendered capture-window times via `datetime.fromtimestamp(...).strftime("%H:%M:%S")` in the analyst's local timezone and without a date, so a ~24h capture could read "at 16:25:31 through 16:24:25" (end before start). Now rendered as UTC ISO-8601 (`format_ts`).
- **Raw epoch floats in TCP flow evidence**: `--tcp` printed `window=1704067200.123->...` and a bare-seconds duration; now uses `format_ts` (UTC ISO) and `format_duration`.
- **Severity miscoloring**: a detection-rendering block colored only `high`→red / `warning`→yellow and sent everything else (including `critical` and `medium`) to green "INFO" — making a *critical* finding look benign. Severity coloring now routes through canonical `coloring.severity_color()` / `severity_label()` helpers that normalize every severity vocabulary (CRITICAL/HIGH/MEDIUM/WARNING/LOW/INFO, any case) to the correct color so critical/high are always red and medium/warning always yellow.

### Added
- `coloring.severity_color()`, `severity_label()`, `severity_rank()`, `normalize_severity()`: a single canonical mapping from any severity string (the codebase mixes HIGH/MEDIUM/LOW and critical/high/warning/info across modules) to color, displayed label, and rank — so future renderers converge on consistent severity presentation.

### Changed (internal — no output change)
- De-duplicated ~70 copies of helper functions into shared, cached implementations. `utils` now hosts the canonical `is_public_ip` / `is_private_ip` / `is_valid_ip` (replacing 27 + 8 + 6 per-module copies), `shannon_entropy` (8 copies), and `read_ber_length` (6 copies). Notably the old `is_public_ip` copies disagreed — some used `is_global` while `files`/`hostname` used a looser `not (private|loopback|multicast|link_local)` test that misclassified CGNAT (100.64/10) and reserved space as public; all callers now share one IANA-correct, `lru_cache`-backed definition.
- TLS helper consolidation completed: `_coerce_int_list`, `_iter_tls_extensions`, `_tls_extension_type`, `_extract_sni`, `_extract_alpn` now live only in `tls_fingerprints.py` (previously duplicated in both `tls.py` and `ips.py`).
- Removed dead code: 11 never-called functions (across `beacon`, `email`, `files`, `modbus`, `timeline`, `rdp`, `enip`) and ~30 now-unused `import ipaddress` / `import math` / `from functools import lru_cache` statements left over after consolidation, plus a stale commented-out block in `dnp3`. Net ~290 fewer lines with byte-identical analysis output (verified against pristine across `--ips/--dns/--routing/--protocols/--threats/--tcp/--udp/--exfil/--timeline` and the TLS pipeline).
- **TLS version mislabeling**: every TLS 1.3 handshake was reported as "TLS 1.2" because the analyzer read the hello's `legacy_version` field (which RFC 8446 pins to 0x0303) and never consulted the `supported_versions` extension (type 43). The negotiated version is now resolved from that extension for both ClientHello and ServerHello, so the version inventory and legacy-TLS findings reflect reality.
- **JA3/JA4 GREASE filtering**: the GREASE test `(value & 0x0F0F) == 0x0A0A` also stripped non-GREASE cipher/extension values like 0x1a2a, corrupting fingerprints. Now matches only the 16 RFC 8701 values (both bytes equal, low nibble 0xA).
- **JA4 / JA4S non-conformance**: the emitted fingerprints did not follow the FoxIO JA4 spec (unsorted extensions, single cipher, wrong separators, 8- vs 12-hex truncation) and could not match public JA4 databases. Rebuilt to spec: `t<ver><d|i><cipherCount><extCount><alpn>_<sha256[:12] of sorted ciphers>_<sha256[:12] of sorted extensions+sigalgs>` (JA4) and the corresponding JA4S layout. JA3 construction is unchanged.
- **ALPN extraction returned the literal string "ProtocolName"** instead of the protocol value (`h2`, `http/1.1`) because it called `str()` on scapy's `ProtocolName` wrapper instead of reading its `.protocol` field. This corrupted ALPN counts, the SNI→ALPN map, and the JA4 ALPN token.
- **IOC domain matching** used an unanchored substring test, so a short domain IOC (`ic.com`) matched inside unrelated names (`magic.com`). Now requires hostname-label boundaries (subdomains like `sub.evil.com` still match `evil.com`; `notevil.com` and `evil.community` do not).
- **MELSEC/SLMP command codes** were wrong (`--melsec`): Remote Run and Remote Stop were swapped (0x1001 was labeled "Remote Stop" — it is Remote *Run*), Remote Stop (0x1002) was absent entirely, Remote Reset was mis-keyed (0x1801 instead of 0x1006, labeled "Remote Unlatch"), and phantom codes 0x0801/0x1801 were present. Corrected to the canonical SLMP set (RUN 0x1001 / STOP 0x1002 / PAUSE 0x1003 / Latch-Clear 0x1005 / RESET 0x1006). This had inverted the most operationally significant OT signal — a PLC halt vs. a PLC run.
- **NTLM/SMB NTSTATUS table** had the account/password status names (`STATUS_WRONG_PASSWORD`, `STATUS_NO_SUCH_USER`, `STATUS_ILL_FORMED_PASSWORD`, etc.) mis-keyed to the 0xC0000074–0xC000007F range; their real keys are 0xC0000062–0xC000006C (MS-ERREF). This made the NTLM `auth_failure_burst` heuristic both over-count (e.g. 0xC000007F STATUS_DISK_FULL was labeled "LOGON_FAILURE") and under-count (real STATUS_WRONG_PASSWORD 0xC000006A / STATUS_NO_SUCH_USER 0xC0000064 responses were unlabeled and never counted). Relocated to the correct keys.
- JA3/JA4/JA4S construction and TLS hello parsing are now shared in `tls_fingerprints.py` instead of being duplicated between `tls.py` and `ips.py`.

### Added
- **GOOSE forward-injection coverage** (`--goose`): in addition to stNum/sqNum *decreases* (replay/stale frames), the analyzer now flags state *jumps* — a multi-state stNum skip or a new state starting mid-sequence (sqNum > 1) — per IEC 61850-8-1 semantics, as an info-severity "GOOSE State Jump" (possible injected GOOSE or missed frames), de-duplicated per publisher/dataset.
- **MELSEC CPU control detection** (`--melsec`): Remote Stop / Pause / Reset are now surfaced as HIGH-severity findings (PLC halt/reset = ATT&CK for ICS T0816 Device Restart/Shutdown) and Remote Run / Latch-Clear as MEDIUM; previously only Batch/Random *writes* were flagged, so a PLC-halt command produced no anomaly.
- **GOOSE simulation/test-bit detection** (`--goose`): a GOOSE frame with the IEC 61850-8-1 `simulation`/test bit set is flagged (IEDs in Sim mode act on it — a classic GOOSE-injection vector, ATT&CK ICS T0852/T0856), as is `ndsCom=true` (control block not commissioned). These flags were parsed but never surfaced.
- **Modbus FC8 diagnostic-control detection** (`--modbus`): FC8 sub-functions Restart Communications (0x0001), Force Listen-Only Mode (0x0004), and Clear Counters & Diagnostic Register (0x000A) are now flagged HIGH — they halt/disrupt the slave or erase forensic counters (T0816 / anti-forensics). Previously all FC8 sub-functions were lumped into one LOW "Diagnostic/Info" finding.
- **MMS control-operation detection** (`--mms`): the IEC 61850 MMS analyzer now flags Write services HIGH (variable write to an IED, T0855), Delete/file-removal services HIGH (config tampering / anti-forensics, T0872), and File open/read/rename MEDIUM. Previously MMS only labeled these services in command summaries without raising findings.
- **DNP3 broadcast-control + Operate-without-Select detection** (`--dnp3`): control/write/restart sent to a DNP3 broadcast link address (0xFFFD–0xFFFF) is flagged HIGH (actuates every outstation at once); an Operate (FC4) with no preceding Select on the same source/address is flagged MEDIUM (single-stage control / possible Select-Before-Operate bypass).
- **IEC-104 reset-process + clock-sync detection** (`--iec104`): Reset Process (C_RP_NA_1, type 105) is flagged HIGH (RTU reinitialisation, T0816) and Clock Synchronization (C_CS_NA_1, type 103) MEDIUM (clock manipulation corrupts sequence-of-events forensics). The ASDU type table also gained labels for the system commands (interrogation/counter-interrogation/clock-sync/reset/test) that previously showed as bare `ASDU <n>`.
- **CIP File/Program download detection** (`--cip`): a write/control service targeting the CIP File Object (class 0x37) is flagged HIGH — program/firmware download (ATT&CK T0843/T0857) — mirroring the existing safety/security-class write detection.

### Fixed
- **Hostname misattribution from name queries** (`--hostnames`, and anything consuming IP↔hostname findings): a host that *queried* mDNS/LLMNR or NetBIOS (NBNS) for another host's name was incorrectly identified *as* that name. For example, three different clients each querying for `trn-adm15.local` were all labeled `trn-adm15.local`, and an NBNS query for `FILESERVER` mislabeled the querier as FILESERVER. The queried name describes the host being looked up, not the asker. mDNS/LLMNR query-name attribution to the querier is removed, and NBNS name-query requests (opcode 0) are no longer attributed to the sender; legitimate self-identification (mDNS/LLMNR/DNS *responses* and NBNS name *registrations*) is unchanged. On a sample mDNS capture this removed 165 bogus findings (221 → 56) while keeping every legitimate mapping. (Regular DNS, TLS SNI→server, TLS cert→sender, HTTP Host→server, NTLM workstation→client, and the server-banner extractors were audited and confirmed already correct.)
- **Sampled Values false-positive storm** (`--sv`): the "SV Sample Counter Decrease" check fired on every normal `smpCnt` wrap. Per IEC 61850-9-2, `smpCnt` resets to 0 at the start of each sampling cycle (thousands of samples/sec → a wrap roughly once per second), so the old check produced dozens of bogus warnings per second of capture and buried any real signal. It now flags only a *small* backward step (out-of-order or replayed SV frame), excluding the large periodic wrap — renamed "SV Sample Counter Regression."
